## Создание структуры проекта
- Структура проекта должна иметь следующую архитектуру: modules (модули), models (модели), services (сервисы), контроллеры (controllers), а также включать следующие слои: web, domain, datasource.
- Каждый слой является отдельным модулем.
- Каждый модуль может содержать другие модули.
- Слой web должен включать в себя как минимум пакеты model, controller, mapper для взаимодействия с клиентом.
- Слой domain должен включать в себя как минимум пакеты model, service для реализации бизнес-логики приложения.
- Слой datasource должен включать в себя как минимум пакеты model, repository, mapper для реализации работы с данными (например, с базой данных).

## Реализация domain-слоя
- Опиши модель игрового поля в виде целочисленной матрицы.
- Опиши модель текущей игры, у которой есть UUID и игровое поле.
- Опиши интерфейс сервиса, у которого есть методы:
  - Метод получения следующего хода текущей игры алгоритмом Минимакс;
  - Метод валидации игрового поля текущей игры (проверь, что не изменены предыдущие ходы);
  - Метод проверки окончания игры.
- Модели, интерфейсы, реализации должны находиться в разных файлах.

## Реализация web-слоя
- Опиши модели игрового поля, текущей игры.
- Реализуй мапперы domain<->web.
- Реализуй контроллер, имеющий метод POST /game/{UUID текущей игры}, который отправляет текущую игру с обновленным игровым полем пользователем и получает в ответ текущую игру с обновленным игровым полем компьютером.

- Модели, интерфейсы, реализации должны находиться в разных файлах.

## Добавление авторизации
- Добавь пользователей, у которых будет UUID, логин, пароль.
  - Реализуй для них отдельный модуль.
- Реализуй поддержку пользователей на всех слоях.
- Создай модель JwtRequest, у которой будет логин и пароль.
- Создай модель JwtResponse, у которой будет тип, accessToken, refreshToken.
- Создай модель RefreshJwtRequest, у которой будет refreshToken.
- Реализуй класс AccessTokenStrategy, который наследуется от PassportStrategy и добавь:
  - Метод валидации accessToken;
  - Метод генерации accessToken по User, в токен необходимо сохранить информацию о UUID в claims (в claims сохраняют часть информации об объекте для последующего использования);
  - Метод получения claims.
- Реализуй класс RefreshTokenStrategy, который наследуется от PassportStrategy и добавь:
  - Метод валидации refreshToken;
  - Метод генерации refreshToken по User, в токен необходимо сохранить инфорацию о UUID в claims;
  - Метод получения claims.
- Реализуй класс AccessTokenGuard, который наследуется от AuthGuard.
- Реализуй класс RefreshTokenGuard, который наследуется от AuthGuard.
- Создай сервис авторизации, который использует UserService:
  - Метод регистрации, который принимает JwtRequest и возвращает JwtResponse и возвращает факт успешной регистрации.
  - Создай метод обновления accessToken и refreshToken, который принимает refreshToken и возвращает JwtResponse.
- Для авторизации создай отдельный модуль.
- Создай endpoint авторизации пользователя.
- Создай endpoint регистрации пользователя.
- Добавь bearer-авторизацию в Authentication.
- Если валидация пройдена успешна, то в UserIdPrincipal сохрани UUID пользователя.
- Если валидация прошла с ошибкой, то добавь в ответ 401 код и не выполняй запрос.
- Добавь endpoint для получения информации о пользователе c валидацей доступа по accessToken.
- Доступ без авторизации должен быть доступен к endpoint'ам: регистрации, авторизации и обновления accessToken.
